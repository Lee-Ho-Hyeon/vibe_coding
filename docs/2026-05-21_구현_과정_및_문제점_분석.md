# 2026-05-21 구현 과정 및 문제점 분석

## 선수 정보 수집 방식 구현

초기 설계 내용을 바탕으로 실제 선수 데이터를 수집하는 작업을 진행하였다. 선수 정보 제공 사이트를 조사한 결과, https://fcplayer.org 에서 다양한 축구 선수 정보를 제공하고 있음을 확인하였다.  

해당 사이트는 실제 축구 통계 기반 서비스라기보다는 게임 데이터를 기반으로 구성되어 있었기 때문에, 하나의 선수에 대해 여러 종류의 카드가 존재하였고 선수 경력보다는 카드 능력치(스탯) 정보가 중심적으로 제공되고 있었다.  

개발자 도구를 활용하여 데이터 구조를 분석한 결과, 선수 정보는 다음 API https://fcplayer.org/api/cards를 통해 불러오고 있음을 확인하였다.  

또한 각 선수 카드에는 고유 식별 번호(ID)가 존재하였으며, 예를 들어 https://fcplayer.org/api/cards/1568과 같은 방식으로 특정 선수 데이터를 확인할 수 있었다.  

이에 따라 일정 범위의 ID를 순차적으로 호출하여 선수 이름을 자동으로 수집하는 방식으로 구현 방향을 결정하였다. 이후 GPT를 활용하여 `save_player_name_by_api.py` 코드를 생성하였고, 해당 코드를 통해 설정한 범위 내의 선수 데이터를 순차적으로 호출하여 선수 이름만 추출하도록 구현하였다. 이때, 같은 선수의 이름이 여러 번 저장되는 것을 막기 위해 반복 저장은 막았다.

수집된 결과는 `fcplayer_names.csv` 파일로 저장하였다.

## 구현 과정에서 발생한 문제점

하지만 실제 구현 과정에서 몇 가지 문제점이 발생하였다.  

첫 번째 문제는 해당 API에 모든 선수 정보가 존재하지 않는다는 점이었다. 해당 API는 게임 유저들의 사용률이 높은 카드 위주로 데이터가 저장되어 있었기 때문에, 실제 게임 내 존재하는 모든 선수 정보를 얻을 수는 없었다. 따라서 데이터의 완전성이 부족하다는 문제가 발생하였다.  

두 번째 문제는 선수 경력 정보를 자동으로 수집하는 과정에서 발생하였다. 저장된 선수 이름을 기반으로 위키데이터(Wikidata) 및 위키백과(Wikipedia)에서 선수의 국적 및 클럽 경력을 자동으로 불러오는 코드를 구현하여 실행하였다.  

그러나 CSV 파일에 저장된 선수 이름과 위키데이터 및 위키백과에 등록된 선수 이름이 조금이라도 다를 경우, 시스템이 다른 선수로 인식하거나 원하는 선수 정보를 찾지 못하는 문제가 발생하였다. 예를 들어 띄어쓰기, 특수문자, 이름 표기 방식 차이 등에 의해 잘못된 검색 결과가 발생하는 경우가 확인되었다.  

이러한 문제를 통해 단순 이름 기반 검색만으로는 안정적인 선수 정보 매칭이 어렵다는 점을 확인하였으며, 이후 단계에서는 국적, 고유 식별 정보, 추가 메타데이터 등을 함께 활용하는 방향에 대해 재논의가 필요하다고 판단하였다.
