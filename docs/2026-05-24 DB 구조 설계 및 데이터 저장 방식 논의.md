# 2026-05-24 DB 구조 설계 및 데이터 저장 방식 논의

## CSV 기반 데이터 구조 확인

기존 구현 과정에서 클럽 기반으로 수집한 선수 정보를 CSV 파일 형태로 저장하였다. 저장된 주요 데이터는 다음과 같았다.

- qid
- wikidata_name
- display_name
- country
- club_qid
- alias

초기에는 해당 CSV 파일을 직접 검색하는 방식도 고려하였으나, 데이터 양이 증가할 경우 검색 속도 저하 및 관리 복잡성이 커질 수 있다고 판단하였다. 또한 선수 이름, 국적, 클럽 경력, 별칭(alias) 등 다양한 조건 기반 검색 기능을 구현하기 위해서는 DB 기반 구조가 더 적합하다고 판단하였다.

---

## DB 저장 방식 논의

논의 결과, 모든 정보를 하나의 테이블에 저장하는 방식보다는 데이터 특성에 따라 여러 개의 테이블로 분리하는 관계형 데이터베이스(Relational Database) 구조를 사용하는 방향으로 결정하였다.

특히 선수는 다음과 같은 특징을 가지고 있었다.

- 한 선수가 여러 클럽 경력을 가질 수 있음
- 한 선수가 여러 alias를 가질 수 있음
- 한 선수가 복수국적(이중국적)을 가질 수 있음

따라서 단일 테이블에 모든 정보를 저장할 경우 데이터 중복 및 관리 문제가 발생할 수 있다고 판단하였다.

---

## Players 테이블 설계

선수 자체 정보를 저장하기 위한 `players` 테이블을 구성하기로 하였다.

저장 항목은 다음과 같다.

- qid
- wikidata_name
- display_name

초기에는 country 정보 또한 players 테이블에 함께 저장하는 방식을 고려하였다. 하지만 복수국적 선수들의 경우 같은 qid를 가진 데이터가 여러 줄 생성되는 문제가 발생하였다.

예를 들어 동일 선수라도 국적에 따라 CSV에 두 줄 이상 저장되는 경우가 있었으며, 이 과정에서 `wikidata_name` 및 `display_name` 표기 또한 조금씩 다르게 저장되는 경우가 존재하였다.

이에 따라 이름 정보는 원본 데이터 형태를 최대한 유지하기 위해 players 테이블에 그대로 저장하기로 결정하였다.

---

## 국적 정보 분리 논의

복수국적 선수 처리를 위해 국적 정보는 별도의 테이블로 분리하기로 결정하였다.

`player_countries` 테이블에는 다음 정보를 저장하도록 설계하였다.

- player_qid
- country

이를 통해 한 선수에게 여러 국적 정보를 연결할 수 있도록 하였으며, 이후 국가별 선수 검색 기능 또한 효율적으로 구현할 수 있도록 설계하였다.

---

## Alias 정보 분리 논의

사용자 검색 편의성을 위해 alias 정보 또한 별도의 테이블로 분리하기로 결정하였다.

`player_aliases` 테이블에는 다음 정보를 저장하도록 설계하였다.

- player_qid
- alias

이를 통해 사용자가 전체 이름이 아닌 일반적으로 많이 사용하는 이름이나 별칭만 입력하더라도 선수 검색이 가능하도록 설계하였다.

예를 들어 “호날두”, “CR7” 등의 검색어를 입력하더라도 동일 선수 정보를 찾을 수 있도록 구현 방향을 정리하였다.

---

## 클럽 정보 저장 구조 논의

클럽 정보는 다음 두 개의 테이블로 분리하여 저장하기로 결정하였다.

### clubs
- club_qid
- club_name

### player_clubs
- player_qid
- club_qid

이를 통해 한 선수의 여러 클럽 경력을 저장할 수 있으며, 반대로 특정 클럽에 소속되었던 선수 목록 또한 효율적으로 조회할 수 있도록 설계하였다.

---

## 최종 결정 사항

논의 결과, 최종 시스템은 CSV 파일을 직접 검색하는 방식이 아니라 관계형 데이터베이스 기반으로 구현하기로 결정하였다.

또한 선수, 클럽, 국적, alias 정보를 각각의 특성에 맞게 분리 저장함으로써 데이터 중복을 줄이고, 이후 검색 기능 및 유지보수 확장성을 고려한 구조로 설계 방향을 정리하였다.
